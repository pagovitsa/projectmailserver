# Docker Mailserver

Portable Docker Mailserver setup based on the official `ghcr.io/docker-mailserver/docker-mailserver:latest` image.

This folder is self-contained so you can move the whole mail server from one host to another without losing data, as long as you copy the entire `mailserver/` directory.

On Linux hosts, the `mailserver` service uses `network_mode: host` so Postfix/Dovecot can see the real remote client IPs instead of Docker proxy addresses.

## Goals

- Docker-only deployment
- portable state with bind-mounted folders
- easy domain change through `.env`
- multi-domain and multi-user support
- no antivirus or antispam by default
- Cloudflare DNS automation for mail records
- automatic Let's Encrypt certificates through Cloudflare DNS challenge
- built-in Roundcube webmail behind a localhost-only HTTP port for nginx proxying

## Folder layout

- `compose.yaml`: the mail server service definition
- `.env`: host/domain specific values for each deployment
- `mailserver.env`: docker-mailserver feature flags
- `domains.txt`: all email domains hosted by this server, one per line
- `config/`: account files, aliases, DKIM data, and other DMS config state
- `data/mail-data/`: mailbox contents
- `data/mail-state/`: Dovecot, Postfix, Redis, and related runtime state
- `data/mail-logs/`: mail logs
- `config/roundcube/`: persistent Roundcube configuration generated by the container
- `data/roundcube/`: Roundcube SQLite DB, temp files, and optional plugin data
- `certs/live/`: active PEM files copied from Certbot for docker-mailserver
- `letsencrypt/`: Certbot account, renewal, work, and log state
- `secrets/`: local-only secret files such as the Cloudflare API token
- `backups/`: optional local tar backups

## First setup

Ο βασικός τρόπος εγκατάστασης είναι πλέον ένα μόνο script:

```bash
./init.sh
```

Το script:

- ρωτάει διαδραστικά για domain, timezone, Roundcube localhost port και Cloudflare token, χωρίς να ρωτάει για staging/dev mode, postmaster email, `mail.<domain>`, `webmail.<domain>`, Cloudflare zone ή πρώτο mailbox όταν χρησιμοποιείς το standard setup
- κρατά prompt για mail/webmail hostname μόνο αν βρει ήδη non-standard τιμή που διαφέρει από το απλό `mail.<domain>` / `webmail.<domain>`
- δημιουργεί μόνο του τα `.env`, `domains.txt` και `secrets/cloudflare.ini`
- ανιχνεύει αυτόματα public IPv4 και IPv6 από το σύστημα και τα γράφει στο `.env` του τρέχοντος host
- γράφει αυτόματα `POSTMASTER_ADDRESS=postmaster@<primary-domain>`
- δημιουργεί αυτόματα το πρώτο mailbox ως `postmaster@<primary-domain>` και εμφανίζει έναν generated αρχικό κωδικό
- τρέχει DNS sync, παίρνει Let’s Encrypt certificate και σηκώνει τον mailserver
- σηκώνει και Roundcube στο `127.0.0.1:$ROUNDCUBE_HTTP_PORT` για proxy από nginx
- δημιουργεί DKIM keys για όλα τα domains που δεν έχουν ήδη keys και τα δημοσιεύει στο Cloudflare
- χρησιμοποιεί production Let’s Encrypt by default (`CERTBOT_STAGING=0`) και αν βρει υπάρχον staging certificate κάνει re-issue αυτόματα για να το αντικαταστήσει
- αν έχει οριστεί `WEBMAIL_HOSTNAME`, το προσθέτει αυτόματα και στα certificate SANs

1. Copy `.env.example` to `.env`.
2. Adjust `MAILSERVER_HOSTNAME`, `PRIMARY_DOMAIN`, and `TZ`. Set `MAILSERVER_ZONE` only when it differs from `PRIMARY_DOMAIN`.
3. Set `MAILSERVER_IPV4` and/or `MAILSERVER_IPV6` in `.env`.
4. Put your hosted mail domains into `domains.txt`, one per line.
5. Create `secrets/cloudflare.ini` from the example and add your Cloudflare API token.
6. Restrict the file permissions:

```bash
chmod 600 secrets/cloudflare.ini
```

7. Start the full workflow:

```bash
./scripts/start.sh
```

Αν χρησιμοποιείς το `./init.sh`, τα παραπάνω αρχεία και βήματα δημιουργούνται αυτόματα και δεν χρειάζεται να τα πειράξεις χειροκίνητα.

This does three things in order:

- syncs the core DNS records to Cloudflare
- obtains or reuses a Let's Encrypt certificate via `dns-01`
- starts `mailserver` and the background renewal container

Because the mailserver runs in host networking mode, ports `25`, `465`, `587`, and `993` must be free on the host itself.

8. If you skip `./init.sh` and do the setup manually, add the first mailbox:

```bash
./scripts/dms.sh email add admin@example.com
```

9. List all mailboxes:

```bash
./scripts/dms.sh email list
```

## Cloudflare token scope

Use a restricted Cloudflare API token with access only to the zones this server manages.

Recommended permissions:

- `Zone:DNS:Edit`
- `Zone:Zone:Read`

If you host multiple domains, include every relevant zone in the token scope.

## Automatic DNS management

The DNS sync script manages these records:

- `A` and/or `AAAA` for `MAILSERVER_HOSTNAME`
- `A` and/or `AAAA` for `WEBMAIL_HOSTNAME` when Roundcube webmail is enabled
- `TXT` SPF for `MAILSERVER_HOSTNAME` (HELO/EHLO identity)
- `MX` for each domain in `domains.txt`
- `TXT` SPF for each hosted domain
- `TXT` DMARC for each hosted domain
- `TXT` DKIM για κάθε domain που έχει generated key

Important:

- The mail host record is created with `proxied=false`. Cloudflare proxy must stay off for mail protocols.
- `PTR` is not managed by Cloudflare DNS. You still need to set reverse DNS at your server provider.
- DKIM γίνεται πλέον αυτόματα από τα helper scripts του project.

## Multi-domain support

docker-mailserver supports multiple domains out of the box when using full email addresses.

Examples:

```bash
./scripts/dms.sh email add admin@example.com
./scripts/dms.sh email add hello@another-domain.tld
./scripts/dms.sh alias add info@example.com admin@example.com
```

If you later add a new domain, you usually do not need to rebuild the server. You add the new accounts and update DNS for that domain.

When you change `domains.txt`, run:

```bash
./scripts/sync-dns.sh
./scripts/generate-dkim.sh
```

Το `generate-dkim.sh` δημιουργεί keys μόνο για τα νέα domains που δεν έχουν ήδη DKIM key files, ώστε να αποφεύγεται άσκοπο key rotation στα υπάρχοντα domains.

## Changing server or domain later

### Move to a new server

Copy the whole `mailserver/` folder to the new host, then run:

```bash
./init.sh
```

Important: copy everything, including:

- `.env`
- `mailserver.env`
- `domains.txt`
- `config/`
- `data/mail-data/`
- `data/mail-state/`
- `data/mail-logs/`
- `certs/`
- `letsencrypt/`
- `secrets/`

Then on the new host rerun `./init.sh` and:

1. let it detect and rewrite the new host IPv4/IPv6 into `.env`
2. keep or update domain/hostname values as needed
3. let it keep production certificate mode by default unless you explicitly decide to change `CERTBOT_STAGING` manually later
4. let it rerun the DNS/cert bootstrap safely

This is the recommended path after migration because host-specific values such as IPs and certificate mode can change even when mail data stays the same.

### Change mail hostname or primary domain

Update `.env` and DNS:

- `MAILSERVER_HOSTNAME`
- `PRIMARY_DOMAIN`

Set `MAILSERVER_ZONE` too only if the Cloudflare zone differs from `PRIMARY_DOMAIN`.

If the server IP changed, also update:

- `MAILSERVER_IPV4`
- `MAILSERVER_IPV6`

The easiest way is to rerun `./init.sh` so the new host values are detected and written for you.

Then re-sync DNS and certificate:

```bash
./scripts/start.sh
```

Notes:

- Existing accounts keep their full email addresses. If users move from `old-domain.tld` to `new-domain.tld`, create the new addresses explicitly.
- Existing mailboxes are preserved because they live in `data/` and `config/`.
- Το `POSTMASTER_ADDRESS` γράφεται από το `init.sh` ως `postmaster@<PRIMARY_DOMAIN>`.
- Reverse DNS must still be updated externally at your VPS or dedicated server provider.

## TLS handling

TLS is managed automatically with Certbot and the Cloudflare DNS plugin.

Certbot stores its own renewal state under `letsencrypt/`, and the active certificate files copied for docker-mailserver are kept under `certs/live/`:

- `fullchain.pem`
- `privkey.pem`

If you want to force a fresh issuance manually, run:

```bash
./scripts/request-cert.sh
```

If you are switching from staging to production on the same `CERTBOT_CERT_NAME`, the bootstrap flow forces a fresh production issuance automatically when it detects a staging certificate. Production is the default bootstrap mode now.

If you add `WEBMAIL_HOSTNAME` later, the bootstrap flow also forces a re-issue when the current certificate does not yet cover the requested SANs.

## Roundcube webmail

Roundcube is included in the compose stack and binds only to localhost:

- `http://127.0.0.1:$ROUNDCUBE_HTTP_PORT`

It connects to the local mailserver with:

- IMAP over SSL on `MAILSERVER_HOSTNAME:993`
- SMTP over SSL on `MAILSERVER_HOSTNAME:465`

The intended deployment is to put your host nginx in front of it and proxy a webmail hostname such as `webmail.example.com`.

Example nginx vhost:

```nginx
server {
	listen 443 ssl http2;
	listen [::]:443 ssl http2;
	server_name webmail.example.com;

	ssl_certificate /home/mailserver/certs/live/fullchain.pem;
	ssl_certificate_key /home/mailserver/certs/live/privkey.pem;

	location / {
		proxy_pass http://127.0.0.1:8080;
		proxy_set_header Host $host;
		proxy_set_header X-Real-IP $remote_addr;
		proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
		proxy_set_header X-Forwarded-Proto $scheme;
	}
}
```

If you change the Roundcube localhost port in `.env`, update the nginx `proxy_pass` target accordingly.

## Git-safe usage

This project is designed to be safe for git when used as a template/bootstrap repository.

Safe to commit:

- `admin.sh`
- `.env.example`
- `compose.yaml`
- `init.sh`
- `mailserver.env`
- `README.md`
- `scripts/`
- `secrets/cloudflare.ini.example`

Do not commit:

- `.env`
- `domains.txt`
- `secrets/cloudflare.ini`
- `config/`
- `data/`
- `certs/`
- `letsencrypt/`
- `backups/`

Recommended workflow:

```bash
git clone <your-repo-url>
cd mailserver
./init.sh
```

Τα παλιά command-style subcommands παραμένουν μόνο για backward compatibility, αλλά το προτεινόμενο flow είναι πλέον το numeric menu.

Για ασφάλεια, η αφαίρεση domain:

- δεν επιτρέπει διαγραφή του `PRIMARY_DOMAIN`
- σταματά αν υπάρχουν mailbox users στο domain
- σταματά αν υπάρχουν aliases που χρησιμοποιούν το domain

## Backup and restore

Create a compressed backup:

```bash
./scripts/backup.sh
```

Restore by unpacking a backup archive over the same folder structure on a stopped stack, then start the service again.

## DNS checklist

- `A` or `AAAA` for `MAILSERVER_HOSTNAME` is automated
- `SPF` TXT for `MAILSERVER_HOSTNAME` (HELO/EHLO) is automated
- `MX` for each hosted domain pointing to `MAILSERVER_HOSTNAME` is automated
- `PTR` matching the mail host
- `SPF` record for sending hosts is automated
- `DKIM` record after generating keys is automated
- `DMARC` record for policy and reporting is automated

## DKIM

Το setup πλέον αυτοματοποιεί και το DKIM:

- το `init.sh` δημιουργεί αυτόματα το `postmaster@<primary-domain>` με generated password και μετά τρέχει DKIM generation για όλα τα domains του `domains.txt`
- τα generated αρχεία αποθηκεύονται στο `config/opendkim/keys/`
- τα DKIM TXT records δημοσιεύονται αυτόματα στο Cloudflare

Αν αργότερα προσθέσεις νέο domain ή νέο zone, τρέξε:

```bash
./scripts/generate-dkim.sh
```
